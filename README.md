# Импорт заказов

Это решение тестового задания. Оно делает три вещи:

1. Читает файл заказов формата `ID товара;ID клиента;Комментарий`.
2. Валидные строки пишет в таблицу `orders`.
3. Невалидные строки пишет в отдельный файл без изменений.

SQL-скрипты лежат отдельно в `database/sql`.

## Исходное задание

Есть база данных для хранения информации о клиентах, товарах и заказах со
следующей структурой:

- `clients (id, name)` - ID клиента и его имя.
- `merchandise (id, name)` - ID товара и его наименование.
- `orders (id, item_id, customer_id, comment, status, order_date)` - ID заказа,
  ID товара, ID клиента, комментарий клиента, статус заказа (`new`, `complete`),
  дата заказа. Структура предполагает, что один заказ - это один товар.

Необходимо:

1. Написать скрипт, который получает на вход текстовый файл с данными о заказах
   с разделителем `;` в формате `ID товара;ID клиента;Комментарий к заказу` и
   загружает содержимое в описанную выше структуру БД. Все невалидные строки
   должны записываться в отдельный файл. Использование сторонних решений /
   библиотек нежелательно.
2. Написать SQL-запросы, возвращающие набор данных по условиям.
   Условие 2a: выбрать имена (`name`) всех клиентов, которые не делали заказы в последние 7 дней.
   Условие 2b: выбрать имена (`name`) 5 клиентов, которые сделали больше всего заказов в магазине.
   Условие 2c: выбрать имена (`name`) 10 клиентов, которые сделали заказы на наибольшую сумму.
   Условие 2d: выбрать имена (`name`) всех товаров, по которым не было доставленных заказов со статусом `complete`.
3. Описать, какие индексы нужны для оптимизации запросов из пункта 2 и почему.

В качестве решения принимается архив, содержащий:

- скрипт импорта заказов из пункта 1;
- набор тестовых данных для скрипта;
- файл с SQL-запросами;
- перечень необходимых индексов с обоснованием.

## Уточнение по `price`

В исходной схеме нет поля для суммы заказа, но пункт 2c требует считать клиентов
с максимальной суммой заказов. Поэтому в решении добавлено минимальное поле
`price` в денежных минимальных единицах:

- `merchandise.price` - текущая цена товара;
- `orders.price` - snapshot цены товара на момент создания заказа.

Это единственное расширение схемы сверх исходного задания.

Почему цена хранится именно в `orders.price`, а не считается из
`merchandise.price` в SQL-запросе:

- `merchandise.price` - это текущая цена товара.
- Заказ должен хранить цену на момент оформления.
- Если завтра цена товара изменится, старые заказы не должны изменить свою сумму.
- Поэтому при импорте текущая цена товара копируется в заказ как snapshot.

Пример: товар стоил `10000`, заказ был создан сегодня, завтра товар стал стоить
`12000`. Сумма старого заказа всё равно должна быть `10000`.

## Что нужно установить

Нужен PHP 8.4.

Для запуска импортёра нужен PDO-драйвер вашей базы:

- для MySQL нужен `pdo_mysql`;
- для PostgreSQL нужен `pdo_pgsql`;
- для локальной проверки на SQLite нужен `pdo_sqlite`.

Для тестов и статического анализа нужен Composer:

```bash
php8.4 $(command -v composer) install
```

Если `composer install` не запускать, тесты и статический анализ работать не
будут, потому что зависимости лежат в `vendor`.

Можно короче:

```bash
make install
```

## Настройка `.env`

Подключение к БД можно задать через `.env`. Для старта скопировать пример:

```bash
cp .env.example .env
```

Поля:

- `DB_DSN` - PDO DSN для подключения к базе.
- `DB_USER` - пользователь БД, если нужен.
- `DB_PASSWORD` - пароль БД, если нужен.
- `IMPORT_INPUT_PATH` - путь к входному файлу заказов.
- `IMPORT_INVALID_PATH` - путь к файлу для невалидных строк.
- `IMPORT_STATUS` - статус создаваемых заказов, обычно `new`.
- `IMPORT_ORDER_DATE` - дата заказа; если пусто, используется текущая дата.

Пример для SQLite:

```dotenv
DB_DSN=sqlite:/absolute/path/to/tests/shop.sqlite
DB_USER=
DB_PASSWORD=
IMPORT_INPUT_PATH=data/orders_input.txt
IMPORT_INVALID_PATH=data/invalid_orders.txt
IMPORT_STATUS=new
IMPORT_ORDER_DATE=2026-06-09
```

Пример для MySQL:

```dotenv
DB_DSN=mysql:host=127.0.0.1;dbname=shop;charset=utf8mb4
DB_USER=shop_user
DB_PASSWORD=secret
IMPORT_INPUT_PATH=data/orders_input.txt
IMPORT_INVALID_PATH=data/invalid_orders.txt
IMPORT_STATUS=new
IMPORT_ORDER_DATE=
```

Параметры командной строки имеют приоритет над `.env`. Например `--dsn` перекроет
`DB_DSN`, а `--input` перекроет `IMPORT_INPUT_PATH`.

## Как запустить импорт

Пример для MySQL:

```bash
php8.4 import_orders.php \
  --dsn="mysql:host=127.0.0.1;dbname=shop;charset=utf8mb4" \
  --user="shop_user" \
  --password="secret" \
  --input="data/orders_input.txt" \
  --invalid="data/invalid_orders.txt"
```

Если `.env` заполнен, можно запускать короче:

```bash
php8.4 import_orders.php
```

Что означают параметры:

- `--dsn` - строка подключения PDO.
- `--user` - пользователь БД, если нужен.
- `--password` - пароль БД, если нужен.
- `--input` - файл с заказами.
- `--invalid` - файл, куда попадут плохие строки.
- `--status` - статус создаваемых заказов, по умолчанию `new`.
- `--date` - дата создаваемых заказов в формате `YYYY-MM-DD`, по умолчанию сегодня.

Что делает импорт по шагам:

1. Открывает входной файл.
2. Читает строки по одной, не загружая весь файл в память.
3. Разбирает строку через разделитель `;`.
4. Проверяет ID товара и ID клиента.
5. Берёт текущую цену товара из `merchandise.price`.
6. Создаёт заказ со статусом `new`, датой заказа и snapshot-ценой.
7. Пишет невалидные строки в отдельный файл.

Все валидные заказы вставляются внутри транзакции. Если во время импорта будет
ошибка БД, транзакция откатится.

## Какая строка считается плохой

Строка не импортируется, если выполняется хотя бы одно условие:

- в строке не ровно три поля через `;`;
- `ID товара` не положительное целое число;
- `ID клиента` не положительное целое число;
- товара с таким ID нет в таблице `merchandise`;
- клиента с таким ID нет в таблице `clients`.

Плохая строка записывается в invalid-файл как есть.

Примеры:

- `1;2;Комментарий` - валидная строка, если товар `1` и клиент `2` есть в БД.
- `abc;2;Комментарий` - невалидная строка, потому что ID товара не число.
- `1;0;Комментарий` - невалидная строка, потому что ID клиента должен быть положительным.
- `1;2` - невалидная строка, потому что полей не три.
- `1;999;Комментарий` - невалидная строка, если клиента `999` нет в БД.

## Быстрая локальная проверка

Создать тестовую SQLite-базу:

```bash
php8.4 -r '$db = new PDO("sqlite:" . __DIR__ . "/tests/shop.sqlite"); $db->exec(file_get_contents(__DIR__ . "/database/sql/schema/sqlite_schema.sql"));'
```

Запустить импорт:

```bash
php8.4 import_orders.php \
  --dsn="sqlite:$(pwd)/tests/shop.sqlite" \
  --input="data/orders_input.txt" \
  --invalid="data/invalid_orders.txt" \
  --date="2026-06-09"
```

Ожидаемый результат:

```text
Processed: 10
Imported: 4
Invalid: 6
```

Проверить, что файл плохих строк совпал с эталоном:

```bash
diff -u data/expected_invalid_orders.txt data/invalid_orders.txt
```

Если `diff` ничего не вывел, всё совпало.

После успешного запуска в тестовой базе будет 4 заказа. В `orders.price` будут
записаны цены из `merchandise.price` на момент импорта:

```text
10000
5000
30000
30000
```

Самый короткий вариант локальной проверки:

```bash
make smoke
```

Эта команда создаст SQLite-базу, запустит импорт и сравнит файл невалидных строк
с эталоном.

## Архитектура

Проект сделан в стиле DDD/CQRS/Hexagonal, но без лишнего фреймворка.

- `import_orders.php` - только CLI-вход и сборка зависимостей.
- `src/Domain` - доменная сущность `Order` и value objects.
- `src/Application` - command use case, handler, ports и mapping входных данных в Entity.
- `src/Infrastructure` - работа с файлами, PDO-репозитории и mapping Entity в SQL-параметры.
- `database/sql` - отдельное хранилище SQL-скриптов: схемы, read-запросы и индексы.
- `tests/Architecture` - тесты, которые не дают слоям смешаться.

Контракты репозиториев лежат в `Application\Port`, а не в `Domain`. Так домен не
знает про способ хранения данных.

Зачем такая структура:

- `Domain` отвечает за смысловые правила: ID положительный, статус допустимый, цена не отрицательная.
- `Application` отвечает за сценарий: импортировать строки, отличить валидные от невалидных, вызвать порты.
- `Infrastructure` отвечает за технические детали: файлы, PDO, чтение SQL-скриптов.
- SQL лежит отдельно, чтобы PHP-код не смешивался с SQL-текстом.

Если нужно поменять SQL под другую СУБД, меняются файлы в `database/sql`, а не
код репозиториев.

## Где SQL

Файлы:

- `database/sql/schema/sqlite_schema.sql` - тестовая SQLite-схема.
- `database/sql/queries/orders_read_queries.sql` - запросы из пункта 2.
- `database/sql/indexes/orders_indexes.sql` - индексы и объяснения.
- `database/sql/persistence/*.sql` - SQL для PDO-репозиториев.

Цена хранится в минимальных денежных единицах. У товара есть текущая цена
`merchandise.price`, а в заказ пишется snapshot этой цены в `orders.price`.
Запрос 2c считает сумму именно по `orders.price`, потому что это историческая
цена конкретного заказа.

Что где смотреть:

- Для проверки структуры таблиц открыть `database/sql/schema/sqlite_schema.sql`.
- Для проверки ответов на пункт 2 открыть `database/sql/queries/orders_read_queries.sql`.
- Для проверки индексов и обоснований открыть `database/sql/indexes/orders_indexes.sql`.
- Для SQL, который используют PDO-репозитории, открыть `database/sql/persistence`.

## Схема

В решении используются три таблицы:

- `clients (id, name)` - клиенты.
- `merchandise (id, name, price)` - товары и текущая цена в минимальных денежных единицах.
- `orders (id, item_id, customer_id, comment, status, order_date, price)` - заказы и snapshot цены товара на момент заказа.

`orders.item_id` ссылается на `merchandise.id`. `orders.customer_id` ссылается
на `clients.id`. Статус заказа ограничен значениями `new` и `complete`.

Поле `price` хранится как integer. Это минимальные денежные единицы, например
копейки или центы. Так проще избежать ошибок округления, которые бывают при
хранении денег в дробных числах.

## Индексы

Индексы лежат в `database/sql/indexes/orders_indexes.sql`.

- `idx_orders_customer_date ON orders (customer_id, order_date)` нужен для поиска клиентов без заказов за последние 7 дней.
- `idx_orders_customer_date` также помогает группировке заказов по клиенту в запросе топ-5 по количеству заказов.
- `idx_orders_customer_price ON orders (customer_id, price)` нужен для суммы заказов по клиенту.
- `idx_orders_item_status ON orders (item_id, status)` нужен для поиска товаров без доставленных заказов.

Для PostgreSQL вместо `idx_orders_item_status` можно использовать частичный
индекс по `status = 'complete'`; пример есть в SQL-файле.

Почему именно такие индексы:

- В запросе 2a база ищет заказы конкретного клиента за период, поэтому нужны `customer_id` и `order_date`.
- В запросе 2b база группирует заказы по клиенту, поэтому индекс с первым полем `customer_id` тоже полезен.
- В запросе 2c база считает сумму по клиенту, поэтому индекс `(customer_id, price)` помогает читать нужные поля компактнее.
- В запросе 2d база проверяет, есть ли у товара заказ со статусом `complete`, поэтому нужны `item_id` и `status`.

Первичные ключи `clients.id`, `merchandise.id` и `orders.id` отдельно индексировать
не нужно: они уже индексируются как primary key.
